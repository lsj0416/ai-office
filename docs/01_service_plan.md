# AI Office — 서비스 기획서

> **"내 맘대로 AI 오피스"** — 역할 기반 멀티 에이전트를 시각적으로 구성하고, 1인 개발자가 실제로 쓰는 AI 팀

---

## 1. 서비스 개요

### 배경 및 문제 정의

1인 개발자 / 소규모 스타트업은 개발 이후 단계에서 막힌다.

- 서비스는 만들었는데 런칭, 마케팅, 운영, 수익화를 혼자 감당해야 함
- 각 영역의 전문가를 고용하기엔 비용이 너무 큼
- 기존 AI 도구(ChatGPT 등)는 우리 회사 맥락을 모름 → 매번 배경 설명 반복

### 핵심 아이디어

> 회사 정보를 입력하면 AI 직원팀이 자동 생성되고,
> 사무실 공간에서 직원들이 실제로 일하듯 움직이며,
> 쓸수록 우리 회사 히스토리를 아는 AI팀이 되는 서비스

### 페르소나

**메인 타겟: 1인 개발자 (= 세종씨 본인)**

```
백엔드 개발자
사이드 프로젝트 여러 개 동시 진행 중
개발 이후 마케팅 / 운영 / 수익화가 막막
툴이 너무 많아서 업무가 파편화됨
전문가 고용 비용이 부담됨
```

**확장 타겟**
- 10인 이하 소규모 스타트업
- 1인 크리에이터 / 유튜버
- 소규모 마케팅 대행사
- 전문가 없는 초기 팀 (디자이너, 마케터 없는 개발팀 등)

## 2. 핵심 기능

### 2-1. 회사 생성

회사 이름 + 비전 + 핵심 비즈니스 입력 → AI가 자동으로 조직 구성

**자동 생성되는 것들**
- 서비스 도메인에 맞는 역할 추천 (PM, 개발자, 마케터, 도메인 전문가 등)
- 각 역할별 페르소나 (회사 맥락이 주입된 시스템 프롬프트)
- 첫 번째 할 일 목록 제안

**예시 — 예비자(교회 예배 준비 서비스) 기준**
| 역할 | 담당 업무 |
|------|----------|
| PM 지민 | 스프린트 관리, 일정 조율 |
| 개발자 태양 | 기술 스펙, 코드 리뷰 |
| 신학자문 요셉 | 성경 본문 검토, 저작권 자문 |
| 마케터 유나 | 교회 커뮤니티 홍보, 런칭 전략 |
| 리뷰어 민준 | QA, 결과물 피드백 |

### 2-2. AI 오피스 공간 (시각적 사무실)

**현재 버전 (HTML 카드 UI)**
- 역할별 책상 카드
- 현재 작업 상태 표시 (말풍선)
- 클릭 → 해당 직원과 채팅

**다음 버전 (Pixi.js 기반 2D 공간)**
- 타일맵 사무실 맵
- 세종씨 캐릭터 WASD 조작
- 직원들 출퇴근 애니메이션
- 회의 시작 → 직원들이 회의실로 이동
- 공간별 다른 대화 톤 (탕비실 = 캐주얼, 발표실 = 프레젠테이션 모드)

**공간 구성**
```
1층  업무공간 (개발/기획/비즈니스 구역)
2층  회의실 / 발표실
3층  탕비실 / 휴게공간
옥상  브레인스토밍 존
지하  서버실 (개발자 전용)
```

### 2-3. 실행 모드 3가지

| 모드 | 설명 | 적합한 상황 |
|------|------|------------|
| **AUTO** | 입력 하나로 여러 역할이 병렬 처리 → 결과 합산 | 빠르게 결과만 원할 때 |
| **PIPELINE** | 역할들이 순서대로 이어받아 점진적 완성 | 단계적 품질 중시 |
| **CHAT** | 사용자가 중간에 개입하며 각 역할과 대화 | 세밀하게 컨트롤 |

### 2-4. 컨텍스트 축적 (쓸수록 똑똑해지는 AI팀)

단순 API 호출이 아닌, 우리 회사 히스토리를 아는 AI팀으로 발전

**축적되는 데이터**
- 사내 메신저 대화
- 문서 (기획서, 보고서 등)
- 회의록 (STT 자동 변환 + 요약)
- 작업 결과물 및 피드백

**RAG 기반 검색 주입**
- 전체 히스토리를 매번 넣지 않음
- 현재 작업과 관련된 기억만 꺼내서 주입
- 비용 90% 절감 + 속도 유지

**계층형 요약**
```
날것 대화 → 일별 요약 → 주별 요약 → 월별 요약
                                        ↑
                                  AI에게 주입
```

### 2-5. 음성 회의

- 미팅룸 입장 → 음성 자동 녹음 (STT: Whisper)
- 역할마다 다른 목소리 (TTS: ElevenLabs)
- 회의 종료 시 자동 요약 + 액션 아이템 추출 + 태스크 등록

## 3. 사용 플로우

### 온보딩 플로우
```
1. 회사 정보 입력 (이름 + 비전 + 비즈니스)
2. AI가 조직 구성 추천
3. 역할 커스텀 or 그대로 수락
4. 사무실 입장 → guided intro로 첫 상호작용 시작
```

### 일반 업무 플로우
```
사무실 입장
→ 직원 책상 클릭 or 말 걸기
→ 작업 요청 (런칭 전략 짜줘 / 이 코드 리뷰해줘 등)
→ 결과물 받기
→ 피드백 → 개선 반복
```

### 회의 플로우
```
회의 시작 버튼
→ 직원들이 회의실로 이동
→ 안건 입력 or 음성으로 자유 진행
→ AI들이 각자 역할에 맞는 의견 제시
→ 종료 시 회의록 자동 생성
→ 액션 아이템 → 태스크로 자동 등록
```

## 6. 디자인 실행 기준

### 6-1. 워크스페이스 정보 구조

AI Office의 기본 홈은 대시보드가 아니라 **오피스**다. 사용자는 "AI 팀이 실제 공간 안에서 일하고 있다"는 감각을 먼저 받아야 한다.

**글로벌 내비게이션**
- 오피스: 기본 홈, 팀 상태 확인과 즉시 상호작용
- AUTO: 팀에게 목표를 던지고 결과를 받는 실행 콘솔
- 채팅: 특정 에이전트와 긴 문맥을 이어가는 1:1 대화
- 회의: 음성 회의와 다중 에이전트 토론
- 팀: 에이전트 구성, 역할, 페르소나 관리
- 태스크: 액션 아이템 정리와 상태 추적
- 설정: 회사 맥락, 제품 정보, 팀 기준값 수정

**정보 구조 다이어그램**
```
워크스페이스 진입
└─ 오피스 (기본 홈)
   ├─ 브랜딩 월: 회사 비전, 운영 톤, 상태 인지
   ├─ 워크 베이: 직원 탐색, 근접 대화, 상태 확인
   ├─ 미팅 존: 회의 소집, 실행 결과 확인
   └─ 팬트리 라운지: 캐주얼 톤 상호작용

보조 작업 화면
├─ AUTO: 목표 입력 → 계획 확인 → 결과 스트림 → 후속 작업 제안
├─ 채팅: 에이전트 선택 → 스레드 로드 → 메시지 스트림
├─ 회의: 음성 입력 → STT → 에이전트 응답 → TTS
├─ 팀: 에이전트 목록 → 생성/수정/정렬
├─ 태스크: 칸반 → 생성/수정/상태 변경
└─ 설정: 회사 정보 → 제품/지표 → 저장
```

**화면 간 이동 원칙**
- 첫 진입과 재방문 모두 오피스를 기본 홈으로 유지한다.
- 오피스는 감각적 홈, 나머지 화면은 실용적 운영 콘솔로 역할을 분리한다.
- 같은 작업을 두 군데에서 중복 수행하지 않는다.
- 오피스에서 시작된 상호작용은 필요할 때만 채팅/AUTO/회의 화면으로 넘긴다.

### 6-2. 온보딩과 첫 30초

워크스페이스 생성 직후에는 `/chat`이 아니라 **오피스 + guided intro**로 진입한다.

**첫 30초 목표**
- 사용자가 "팀이 생성됐다"는 사실을 시각적으로 이해한다.
- 무엇을 먼저 해야 하는지 고민하지 않는다.
- 첫 성공 경험을 1분 안에 만든다.

**guided intro 순서**
```
워크스페이스 생성 완료
→ 오피스 진입
→ 상단 1회성 안내: "직원에게 다가가서 첫 대화를 시작하세요"
→ 첫 추천 액션 3개 제시
   1. PM에게 오늘 목표 묻기
   2. 첫 회의 소집하기
   3. 첫 태스크 만들기
→ 첫 행동 완료 후 안내 축소
```

### 6-3. 핵심 화면별 상태 규칙

빈 상태는 메시지가 아니라 **다음 행동 CTA**여야 한다.

| 기능 | LOADING | EMPTY | ERROR | SUCCESS | PARTIAL |
|------|---------|-------|-------|---------|---------|
| 워크스페이스 목록 | 스켈레톤 리스트 | "첫 회사를 만들어 팀을 시작하세요" + `새 워크스페이스` | 재시도 버튼 + 로그인 상태 안내 | 최근 생성 순 목록 | 일부 필드 누락 시 비전 없이 카드 표시 |
| 오피스 | "오피스 캔버스를 재구성하는 중..." | "직원이 없습니다" + `팀 탭으로 이동` | 캔버스 로드 실패 + `새로고침` | 구역/직원/상호작용 힌트 표시 | 일부 직원만 로드되면 로드된 직원부터 표시 |
| 오피스 대화 | 스레드 로드 중 표시 | "가까이 가서 첫 대화를 시작하세요" | "응답 중 오류가 발생했습니다" + 다시 보내기 | 스트리밍 응답 완료 + 메모리 저장 | 스레드만 있고 메시지 없으면 추천 질문 3개 노출 |
| AUTO | 실행 계획 준비 중 | "팀에게 목표를 던져보세요" + 예시 프롬프트 | 실행 실패 + 재시도 | 계획, 결과, 후속 작업 제안 노출 | 일부 에이전트 응답 중이면 현재 진행 에이전트 강조 |
| 회의 | 참가자 집합 중 | "직원을 1명 이상 선택하세요" + 회의 목적 예시 | 마이크 권한 거부, STT/TTS 실패별 분리 안내 | 회의 결과와 액션 아이템 제시 | 일부 참가자 대기 중이면 집합 상태 유지 |
| 태스크 | 칸반 스켈레톤 | "첫 태스크를 만들어 팀의 다음 행동을 정리하세요" + `태스크 추가` | 저장 실패, 삭제 실패를 인라인 경고 | 칸반 3열과 완료 수치 갱신 | 담당자 없는 태스크는 `미배정` 상태로 표시 |
| 설정 | 폼 로드 중 | 해당 없음 | 저장 실패 메시지 + 입력 보존 | "저장되었습니다" 토스트/인라인 피드백 | 일부 선택 정보만 비어 있어도 저장 가능 |

### 6-4. 사용자 여정과 감정선

| 단계 | 사용자 행동 | 사용자 감정 | 화면이 해야 할 일 |
|------|-------------|-------------|--------------------|
| 1 | 회사 생성 | 기대, 약간의 불안 | 입력 수를 최소화하고 생성 직후 팀이 생겼음을 보여준다 |
| 2 | 첫 오피스 진입 | 신기함, 탐색 욕구 | guided intro로 첫 행동을 밀어준다 |
| 3 | 첫 직원 대화 | 시험해보는 마음 | 답변 속도와 캐릭터 개성을 동시에 보여준다 |
| 4 | 첫 회의/AUTO 실행 | 일 맡기는 재미 | 팀이 병렬로 일하는 장면과 결과를 연결한다 |
| 5 | 태스크/설정 정리 | 실무 전환 | 감성보다 실행 편의가 앞서도록 운영 콘솔을 제공한다 |
| 6 | 재방문 | 익숙함, 속도 기대 | 오피스는 현재 상태 요약, 콘솔은 즉시 실행 중심으로 간다 |

**시간축 원칙**
- 5초: 브랜드와 공간이 기억나야 한다.
- 5분: 첫 대화, 첫 회의, 첫 태스크 중 하나를 끝낼 수 있어야 한다.
- 장기: 단순 챗봇이 아니라 "우리 팀"이라는 느낌이 쌓여야 한다.

### 6-5. UI 톤과 AI Slop 방지 규칙

AI Office는 **두 레이어**로 운영한다.

- `workspace` 레이어: 차분한 운영 콘솔
- `office` 레이어: 몰입형 픽셀 공간

둘은 다른 표정이지만 같은 제품이어야 한다.

**운영 원칙**
- 오피스는 몰입형 홈이다. 세계관과 팀 존재감을 책임진다.
- AUTO, 채팅, 태스크, 설정은 운영 콘솔이다. 빠른 읽기와 실행이 우선이다.
- 오피스 밖 화면도 동일한 색 토큰과 정보 밀도를 유지해 같은 제품처럼 보이게 한다.
- 평범한 SaaS 카드 격자, 과한 데코레이션, 목적 없는 그라데이션을 기본 패턴으로 쓰지 않는다.
- "깔끔한 현대적 UI" 같은 추상 표현 대신, 화면의 역할과 우선순위를 구체적으로 서술한다.

**금지 패턴**
- 첫 화면을 일반적인 카드 대시보드로 시작하는 것
- 모든 작업 화면을 흰 카드와 파란 버튼만으로 퉁치는 것
- 모바일에서 데스크톱 탐험 UX를 그대로 축소하는 것

### 6-6. 디자인 시스템 정렬

현재 구현 기준으로 재사용해야 하는 디자인 자산은 다음과 같다.

- `workspace-*` 토큰: 운영 콘솔용 배경, 라인, 표면, 강조 색상
- `office-*` 토큰: 픽셀 오피스용 패널, 텍스트, 인터랙션 색상
- 픽셀 폰트 `FS Pixel Sans`: 공간형 UI와 간판, 상태 라벨에만 선택적으로 사용
- 워크스페이스 셸: 좌측 내비 + 모바일 전용 내비 분기
- 오피스 존 카피: 브랜딩 월, 워크 베이, 미팅 존, 팬트리 라운지

`DESIGN.md`는 아직 없다. 디자인 시스템 문서가 없으므로 구현 전에 `/design-consultation`으로 명시적 시스템 문서를 만드는 것을 권장한다.

### 6-7. 반응형과 접근성 기준

**반응형**
- 데스크톱: 오피스 캔버스 + 우측 채팅 패널 2열 구조
- 태블릿: 오피스 중심, 보조 패널 폭 축소
- 모바일: 오피스 화면은 유지하되 조작은 탭 중심으로 단순화
- 모바일 내비: 하단 4탭 고정 + 나머지는 More sheet로 제공

**오피스 조작 원칙**
- 데스크톱: WASD/방향키 이동, `E` 상호작용, `R` 회의 소집, `ESC` 닫기
- 모바일: 자유 이동보다 캐릭터 탭, 바텀시트, 빠른 액션 중심

**접근성**
- 터치 타깃 44px 이상
- 색상만으로 상태를 구분하지 않고 라벨을 함께 제공
- 키보드로 닫기, 이동, 주요 액션 실행 가능
- 로딩/오류/성공 메시지는 텍스트로 명시
- 스크린리더용 랜드마크와 버튼 라벨을 제공
